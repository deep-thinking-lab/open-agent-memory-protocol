## 概要

OAMP v1.1 は v1.0.0 に対して **厳密に加算的** なマイナーバージョンです。これは、v1.0 が意図的に「将来の考慮事項」として延期した二つの OPTIONAL 機能を定義します：

- クライアントが `KnowledgeEntry` と `UserModel` イベントをリアルタイムで受信できる **ストリーミングサブスクリプショントランスポート**。
- クライアントが過去の時点でのメモリ状態を照会できる **バイテンポラル `as_of` クエリパラメータ**。

v1.1 準拠のバックエンドは、依然としてすべての v1.0 要件を満たさなければなりません。新しい機能は、小さな **機能発見エンドポイント** を通じて広告されるため、v1.0 クライアントは相互運用可能です。v1.1 は破壊的なスキーマやエンドポイントの変更を導入せず、v1.0 クライアントは v1.1 バックエンドとワイヤ互換性を保ちます。

これらの機能を「v2.0 スコープ」から v1.1 OPTIONAL に昇格させる動機は実用的です：リファレンス実装（cosmictron、kizuna-mem）は、役立つ製品表面を提供するために両方の機能を必要とし、これらの機能に対する OPTIONAL スペックが存在しないと、エコシステムが整合する前に互換性のないベンダー拡張が生じます。

この文書における「MUST」、「MUST NOT」、「REQUIRED」、「SHALL」、「SHALL NOT」、「SHOULD」、「SHOULD NOT」、「RECOMMENDED」、「MAY」、「OPTIONAL」というキーワードは、[RFC 2119](https://www.ietf.org/rfc/rfc2119.txt) に記載されている通りに解釈されるものとします。

## 1. v1.0 との関係

v1.1 は、v1.0 のスキーマ、エンドポイント、要件、セマンティクスをすべて変更なしに再利用します。§3 と §4 の追加のみが新しいものです。ドキュメントは、v1.1 専用フィールドを使用する場合にのみ `oamp_version` を `"1.1.0"` に設定するべきであり、それ以外の場合は `"1.0.0"` が正しく、ポータビリティのために推奨されます。

v1.1 バックエンドは、`oamp_version` が `"1.0.0"` または `"1.1.0"` のいずれかであるドキュメントを受け入れなければなりません。v1.0 バックエンドは、理解できない v1.1 専用フィールドを含む `"1.1.0"` ドキュメントを拒否しなければなりません（v1.0 §10.2 に従ったメジャーバージョン互換性ルール）。ただし、v1.1 が新しいトップレベルの必須フィールドを導入していないため、v1.0 バックエンドは、v1.0 フィールドのみを含む `"1.1.0"` ドキュメントを受け入れるべきです。

## 2. 機能発見

v1.1 は、クライアントがバックエンドがサポートする OPTIONAL 機能を発見できる単一の新しいエンドポイントを導入します。

### 2.1 GET `/v1/capabilities`

バックエンドのプロトコルサーフェスを説明する JSON オブジェクトを返します。

**レスポンス:**

```json
{
  "oamp_version": "1.1.0",
  "capabilities": {
    "streaming": {
      "supported": true,
      "subprotocol": "oamp.v1",
      "endpoint": "/v1/stream",
      "event_types": ["knowledge_created", "knowledge_updated",
                      "knowledge_deleted", "user_model_updated"]
    },
    "as_of": {
      "supported": true,
      "endpoints": ["/v1/knowledge", "/v1/knowledge/{id}",
                    "/v1/user-model/{user_id}"],
      "min_resolution_ms": 1
    },
    "user_id_format": {
      "description": "テナント:ノードの複合 (例: '1:user')",
      "pattern": "^[0-9]+:.+$"
    },
    "id_preservation": "preserved",
    "content_types": ["application/json", "application/protobuf"],
    "auth_schemes": ["bearer"]
  }
}
```

**要件:**

- v1.1 を広告するバックエンドは、このエンドポイントを実装しなければなりません。
- すべての `capabilities.*.supported` フィールドはブール値でなければなりません。
- クライアントは、このエンドポイントを接続のライフタイム中に最大1回呼び出し、結果をキャッシュするべきです。
- バックエンドは、`capabilities.metadata`（オブジェクト）の下にベンダー固有のキーを含めることができます。クライアントは未知のキーを許容しなければなりません。

**`user_id_format` (REQUIRED):**

バックエンドは、複数の OAMP バックエンドを橋渡しするクライアントがクロスバックエンドのインポート/エクスポートを試みる前に互換性を事前確認できるように、`user_id` エンコーディングフォーマットを広告しなければなりません。このオブジェクトには以下が含まれます：

| フィールド | 型 | 要件 | 説明 |
|-------|------|-------------|-------------|
| `description` | string | MUST | フォーマットの人間可読な説明 (例: `"テナント:ノードの複合 (例: '1:user')"`, `"64文字の小文字の16進数 Ed25519 公開鍵"`)。 |
| `pattern` | string | MAY | このバックエンドの有効な `user_id` 値に一致する ECMA-262 正規表現。クライアントはこれを事前検証に使用することができます。 |

OAMP ドキュメント内の `user_id` フィールドは不透明な文字列のままであり（スキーマ内にフォーマット制約はありません）、機能広告はクライアント側の互換性チェックのためのものです。互換性のない `user_id` フォーマットを持つバックエンドを橋渡しするクライアントは、クロスバックエンド転送中に `user_id` 値を変換しなければなりません（これはクライアントの責任であり、バックエンドの責任ではありません）。

**`id_preservation` (REQUIRED):**

クライアントが提供したエントリ ID を `POST /v1/import` 中にバックエンドが保持するかどうかを示す文字列です。以下のいずれかです：

- `"preserved"` -- バックエンドはクライアントが提供した `id` を変更せずに保存し、返します。インポートレスポンスの `id_mappings` フィールドは常に空 `{}` になります。
- `"regenerated"` -- バックエンドはインポートされたエントリに新しい ID を割り当てることができます（例：内部キーからの決定論的導出）。インポートレスポンスの `id_mappings` フィールドは、各元の ID から新しく割り当てられた ID へのマッピングを含まなければなりません。

複数の OAMP バックエンドを橋渡しし、エントリ ID を結合キーとして使用するクライアントは、`id_preservation` を確認し、もし `"regenerated"` であれば、インポートレスポンスから `id_mappings` を適用して参照の整合性を維持しなければなりません。

### 2.2 v1.0 後方互換性

v1.0 バックエンドは `/v1/capabilities` に対して `404 Not Found` を返します。クライアントはこのレスポンスを「バックエンドは v1.0 であり、OPTIONAL 機能は利用できない」と見なし、REST のみの動作にフォールバックしなければなりません。v1.0 バックエンドは `user_id_format` や `id_preservation` を広告しないため、複数のバックエンドを橋渡しするクライアントはこれを適切に処理しなければなりません（§5 を参照）。

### 2.3 インポートレスポンスの形状 (v1.0 §6.4 の明確化)

v1.0 §6.4 は `POST /v1/import` が「`200 OK` と要約を返す」と定義しましたが、ステータスコードやレスポンスボディの形状を固定していません。v1.1 は以下を義務付けます：

**ステータスコード:** `201 Created`。（クライアントは、後方互換性のために v1.0 バックエンドからの `200 OK` も受け入れなければなりません。）

**レスポンスボディ:**

```json
{
  "imported": 5,
  "skipped": 0,
  "rejected": 0,
  "id_mappings": {
    "88f88510-928b-49d9-aff1-4f32acbf1f97": "a299eeae-39ac-4248-ae24-007302cb64fc"
  }
}
```

| フィールド | 型 | 要件 | 説明 |
|-------|------|-------------|-------------|
| `imported` | integer | MUST | 成功裏にインポートされたエントリの数。 |
| `skipped` | integer | MUST | スキップされたエントリの数（例：同等または高い信頼度の重複）。 |
| `rejected` | integer | MUST | 検証エラーにより拒否されたエントリの数。 |
| `id_mappings` | object | MUST | 元の ID から割り当てられた ID へのマッピング。すべての ID が保持された場合は空 `{}`。§2.4 を参照。 |
| `rejections` | array | MAY | 拒否されたエントリの詳細。各要素: `{"id": "...", "reason": "..."}`。 |

### 2.4 インポート時のエントリ ID 保持 (v1.0 §4.4 の明確化)

実装はインポート中にエントリ ID を保持または再生成することができますが、インポートレスポンスを通じて結果を伝達しなければなりません：

- **ID保持バックエンド**（機能で `id_preservation: "preserved"` として広告される）：クライアントが提供した `id` を変更せずに保存します。インポートレスポンスの `id_mappings` フィールドは `{}` でなければなりません。
- **ID再生成バックエンド**（機能で `id_preservation: "regenerated"` として広告される）：インポート中に新しい ID を割り当てます（例：決定論的導出）。インポートレスポンスの `id_mappings` フィールドは、インポートされたエントリの元の ID を新しく割り当てられた ID にマッピングしなければなりません。

これにより、どちらのアーキテクチャパターンも変更を必要とせずに対応できます。ID の安定性に依存するクライアント（例：エントリ ID に基づいてメモリグラフを構築するエージェント）は、`id_mappings` を確認し、インポート後に参照を更新しなければなりません。

## 3. ストリーミングトランスポート (OPTIONAL)

### 3.1 動機

OAMP v1.0 はポーリングベースです：クライアントはメモリの変更について再度検索クエリを発行することで学習します。インタラクティブエージェント、可視化サーフェス、ダッシュボードにとって、これは高いポーリング負荷または古い UI を生じさせます。v1.1 は、クライアントが発生するメモリの変化にサブスクライブできる WebSocket サブプロトコルを定義します。

これは OPTIONAL です。なぜなら (a) すべてのバックエンドがリアルタイムイベントソースを持っているわけではなく、(b) v1.0 のポーリングモデルはバッチエージェントにとって正しく十分であるからです。

### 3.2 エンドポイント

ストリーミングサポートを持つ v1.1 バックエンドは、以下を公開しなければなりません：

- **URL:** `wss://{host}/v1/stream`（または非 TLS 開発用の `ws://`）
- **サブプロトコル:** `oamp.v1`（標準 WebSocket の `Sec-WebSocket-Protocol` ヘッダーを介して交渉される）

クライアントがサブプロトコルリストで `oamp.v1` を要求しない場合、バックエンドは HTTP `400 Bad Request` でアップグレードを拒否しなければなりません。

### 3.3 認証

WebSocket アップグレードは、REST API と同じ方法で認証されなければなりません。バックエンドは、アップグレードリクエストの `Authorization` ヘッダーを介してベアラートークンを受け入れるべきであり、WebSocket アップグレードでヘッダーを設定できないブラウザクライアントのために `?token=` クエリパラメータとして受け入れることもできます。選択されたスキームは `/v1/capabilities` で宣言されなければなりません。

### 3.4 フレームフォーマット

すべてのフレームは **テキストフレーム** であり、単一の JSON オブジェクトを運びます。（バイナリフレームは将来の protobuf モードストリーミングのために予約されており、v1.1 では使用されてはなりません。）

すべてのフレームは以下の形状を持ちます：

```json
{
  "oamp_version": "1.1.0",
  "type": "<frame_type>",
  "id": "<uuid_v4>",
  "ts": "<iso8601>",
  "payload": { ... }
}
```

`id` はクライアントが返信を相関させるために使用するフレームごとの識別子です；`ts` はフレームが発信された時点でのバックエンドの単調なタイムスタンプです。

### 3.5 クライアント → サーバーフレーム

| `type`         | 目的                                            |
|----------------|----------------------------------------------------|
| `subscribe`    | フィルターを使用してサブスクリプションを開く。                  |
| `unsubscribe`  | 以前に開いたサブスクリプションを閉じる。            |
| `ping`         | 生存確認プローブ；バックエンドは `pong` で応答しなければならない。  |

**`subscribe` ペイロード:**

```json
{
  "subscription_id": "<client-chosen-string>",
  "user_id": "user-123",
  "event_types": ["knowledge_created", "knowledge_updated"],
  "filters": {
    "category": ["preference", "correction"],
    "tags": ["language"]
  },
  "include_initial_snapshot": false
}
```

- `subscription_id` はクライアントが選択したもので、接続ごとに一意でなければなりません。サーバーは、以降のすべてのイベントとサブスクリプション解除の確認でこれを使用します。
- `user_id` は REQUIRED です。バックエンドはクロスユーザーサブスクリプションを拒否しなければなりません（`"forbidden"` コードの `error` フレームを返す）。
- `event_types` は、バックエンドがサポートするすべてのイベントタイプにサブスクライブするために省略することができます。
- `filters` は OPTIONAL です；認識されたフィルターキーは §3.7 にリストされています。未知のフィルターキーは無視され、拒否されてはなりません。
- `include_initial_snapshot`（デフォルトは `false`）：`true` の場合、バックエンドは、ライブイベントが流れる前に現在の一致する状態を含む `knowledge_snapshot` フレームを一つ発信しなければなりません。

### 3.6 サーバー → クライアントフレーム

| `type`                | 目的                                              |
|-----------------------|------------------------------------------------------|
| `subscribed`          | サブスクリプションを確認する。                          |
| `unsubscribed`        | サブスクリプション解除を確認する。                       |
| `knowledge_created`   | 新しい `KnowledgeEntry` が保存されました。                   |
| `knowledge_updated`   | 既存の `KnowledgeEntry` が修正されました (PATCH)。   |
| `knowledge_deleted`   | `KnowledgeEntry` が永久に削除されました。          |
| `knowledge_snapshot`  | `include_initial_snapshot` のためのワンショットスナップショット。    |
| `user_model_updated`  | `UserModel` 行が更新されました。                       |
| `error`               | プロトコルまたはアプリケーションエラー。                     |
| `pong`                | 生存確認の応答。                                      |

**`knowledge_created` ペイロード:**

```json
{
  "subscription_id": "<echoed-from-subscribe>",
  "entry": { /* 完全な v1.0 KnowledgeEntry ドキュメント */ }
}
```

`knowledge_updated` は **更新後** のエントリを運びます。`knowledge_deleted` は、v1.0 の「ログにコンテンツなし」ルールを満たすために `{ "subscription_id": "...", "id": "<uuid>", "user_id": "..." }` のみを運び、削除されたコンテンツは再放送されてはなりません。

**`error` ペイロード:**

```json
{
  "subscription_id": "<id-or-null>",
  "code": "forbidden | invalid | rate_limited | internal",
  "message": "人間が読み取れる",
  "retryable": false
}
```

### 3.7 認識されたフィルターキー

| キー        | 型            | セマンティクス                                  |
|------------|-----------------|--------------------------------------------|
| `category` | array of string | これらの v1.0 カテゴリのいずれかに一致します。        |
| `tags`     | array of string | エントリは少なくとも一つのリストされたタグを含まなければなりません。|
| `min_confidence` | number    | エントリの `confidence` はこの値以上でなければなりません。 |

バックエンドは追加のフィルターキーをサポートすることがあります；それらは `/v1/capabilities.streaming.filter_keys` で広告されなければなりません。

### 3.8 バックプレッシャーと配信

- プロトコルは **at-most-once** です。クライアントが追いつけない場合、バックエンドはイベントをドロップすることができ、`"rate_limited"` コードの単一の `error` フレームを発信してギャップを示すべきです。正確に一度のセマンティクスが必要なクライアントは、`/v1/knowledge` ポーリングを介して調整しなければなりません。
- バックエンドは、クライアントトラフィックが 60 秒間ない場合（`ping` なし、他のフレームなし）に WebSocket を閉じなければなりません。クライアントは 30 秒ごとに `ping` を送信するべきです。
- バックエンドは、接続ごとに少なくとも 16 の同時サブスクリプションを許容しなければなりません。

### 3.9 プライバシー

v1.0 §8 のプライバシールールは、ストリーミングコンテンツに適用され、REST レスポンスのように扱われます：

- 知識コンテンツは接続のいずれかの側でログに記録されてはなりません。
- `knowledge_deleted` フレームは削除されたコンテンツを含めてはなりません。
- サブスクリプションは単一の `user_id` にスコープされなければなりません。マルチユーザーファンアウトは v2.0 の懸念です。

## 4. バイテンポラル `as_of` クエリパラメータ (OPTIONAL)

### 4.1 動機

多くのメモリバックエンド（cosmictron、kizuna-mem、その他）はすでにバイテンポラルデータを保存しています — `valid_time` 軸（事実が世界で真であった時）と `ingest_time` 軸（システムがその事実を学習した時）。v1.0 には「時刻 T に何を知っていましたか？」と尋ねる方法がなく、これは以下のために必要です：

- エージェントの意思決定のリプレイとデバッグ。
- コンプライアンス監査（「この決定が下されたときにファイルに何がありましたか？」）。
- 可視化ダッシュボードにおける逆時間旅行 UI。

v1.1 は、このストレージ機能を公開する単一の普遍的に適用可能なクエリパラメータを定義しますが、内部表現を指定するものではありません。

### 4.2 パラメータ

`as_of` サポートを持つバックエンドは、以下のエンドポイントで次のクエリパラメータを受け入れなければなりません：

```
?as_of=<iso8601-datetime>
```

影響を受けるエンドポイント：

| エンドポイント                          | `as_of` を使用したセマンティクス                          |
|-----------------------------------|-------------------------------------------------|
| `GET /v1/knowledge?query=...`     | `as_of` の時点で存在していたインデックスを検索します。      |
| `GET /v1/knowledge/{id}`          | `as_of` の時点でのエントリの状態を返します。         |
| `GET /v1/user-model/{user_id}`    | `as_of` の時点でのユーザーモデルを返します。            |

ミューテーションエンドポイント（`POST`、`PATCH`、`DELETE`）は `as_of` を受け入れてはならず、パラメータが提供された場合は `400 Bad Request` で応答しなければなりません。

### 4.3 セマンティクス

二つのセマンティクス軸が可能です。バックエンドはデフォルトで **ingest_time セマンティクス** を選択しなければなりません：「正確に `as_of` で発行された場合にこの同じクエリが返したであろう結果を示してください。」これは唯一の普遍的に明確な解釈であり、すべての既知のリファレンスバックエンドが実装しています。

バックエンドが `valid_time` クエリ（世界状態軸）をサポートする場合、別の明示的に名前付けされたパラメータ（例：`?valid_at=`）を通じてそれらを公開しなければなりません。v1.1 はこの目的のために `valid_at` を予約しますが、標準化は行いません；それは v2.0 の作業です。

### 4.4 レスポンスの形状

レスポンスボディは、同等の v1.0 レスポンスと同一でなければなりません。v1.1 は、ボディが記述する歴史的状態を変更するだけです。

v1.1 対応のバックエンドは、使用したタイムスタンプをエコーするレスポンスヘッダー `OAMP-As-Of: <iso8601>` を含めるべきです。クライアントはこれを使用してタイムスタンプの正規化を検出することができます（例：バックエンドがストレージ解像度に丸めた場合）。

### 4.5 範囲外のタイムスタンプ

- 将来の `as_of` は `now` として扱われなければなりません。バックエンドは `OAMP-As-Of` を実際に解決されたタイムスタンプに設定するべきです。
- ユーザーの最初のインジェストイベントの前の `as_of` は、空の結果セット（HTTP 200）を返さなければならず、404 ではありません。
- バックエンドが保持期限/スナップショットの期限切れにより解決できない `as_of` は、`409 Conflict` を返し、`code: "as_of_expired"` を含まなければなりません。

### 4.6 機能広告

`/v1/capabilities.as_of.min_resolution_ms` は、バックエンドが解決できる最小の時間デルタを報告しなければなりません（例：スナップショット間隔）。クライアントはサブミリ秒の解像度を仮定すべきではありません。

---

## 5. コンプライアンス

**v1.1 準拠** を主張するバックエンドは、以下を満たさなければなりません：

1. すべての v1.0 の必須要件を満たすこと。
2. 真実の機能フラグを返す `GET /v1/capabilities` を実装すること。
3. 広告する各 OPTIONAL 機能（`streaming`、`as_of`）について、§3 または §4 で説明される完全なサーフェスを実装すること。
4. 文書化された HTTP/WebSocket エラーコードでサポートされていない OPTIONAL 機能を拒否し、決して静かに無視してはならない。

バックエンドは、**サポートされる OPTIONAL 機能がゼロ** である v1.1 準拠を主張することができます。これは有用です：それはクライアントに対して、バックエンドが v1.1 の語彙を理解しており、将来の OPTIONAL 機能を `/v1/capabilities` で公開することを示します。

`/validators/validate.sh` のバリデーターは、別の PR で v1.1 のフィクスチャを取得します；v1.1 ドキュメントは、変更なしに v1.0 JSON スキーマに対してバリデートされなければなりません。

---

## 6. v1.0 クライアントの移行パス

v1.0 クライアントが v1.1 バックエンドと通信する場合、変更なしで動作し続けます。v1.0 クライアントが v1.1 機能を選択したい場合：

1. `GET /v1/capabilities` を発行し、レスポンスを確認します。
2. `streaming.supported` の場合、WebSocket を開き、§3 に従います。
3. `as_of.supported` の場合、役立つ場合は読み取りリクエストに `?as_of=` を追加します。

保存されたドキュメントの `oamp_version` を上げる必要はありません。バージョン文字列はドキュメントを説明するものであり、セッションではありません — v1.1 クライアントは v1.0 ドキュメントを完全に保存できます。

## 7. v1.1 最終化のためのオープンクエスチョン

これらは、v1.1 が安定とマークされる前にコミュニティの議論のために追跡されます：

1. **再接続時のサブスクリプションの再開。** クライアントは `subscribe` で `since=<event_id>` を渡して見逃したイベントをリプレイできるべきでしょうか？ これはバックエンドがイベントログを保持する必要があるため、多くはそうではありません。*仮の回答：* v2.0 に任せる。
2. **スナップショットのページネーション。** 100k エントリを持つユーザーのための `knowledge_snapshot` フレームは、現在は単一のフレームです。チャンク化を義務付けるべきでしょうか？ *仮の回答：* バックエンドが `streaming.snapshot_max_entries` 制限を報告する場合は `snapshot_chunk` を義務付け、それ以外は単一フレーム。
3. **`valid_at` の標準化。** 財務やコンプライアンスからの実際の需要は、`valid_at` がいくつかのワークフローにとって `as_of` よりも有用であることを示唆しています。 *仮の回答：* v1.1 に `as_of` を実装し、`valid_at` は v1.2 または v2.0 に保持し、2 つ以上のバックエンドが相互運用可能な実装を出荷するまで待つ。
4. **gRPC ストリーミング。** ストリーミングサブプロトコルは gRPC バインディングを持つべきでしょうか？ *仮の回答：* `/proto/` ディレクトリは、JSON 形状が安定した後に `service Stream { rpc Subscribe(stream SubscribeRequest) returns (stream Event); }` を取得します。

## 付録 A: 機能スキーマ

`/v1/capabilities` レスポンスのための JSON スキーマは、§2 が最終化され次第 `spec/v1.1/capabilities.schema.json` に追加されます。§2.1 の形状が作業定義です。

## 付録 B: リファレンス実装ターゲット

二つのリファレンスバックエンドが、このドラフトと同時に v1.1 OPTIONAL 機能を導入します：

- **cosmictron** (Rust) — `/v1/capabilities`、`/v1/oamp/*` REST、`/v1/oamp/stream` WebSocket サブプロトコル、メモリ読み取り時の `?as_of=`。詳細は `cosmictron/docs/design/OAMP_TRANSPORT.md` を参照。
- **kizuna-mem** (Zig コア + Rust サイドカー) — 同じサーフェス；Rust サイドカーから提供される WebSocket。詳細は `kizuna-dream/docs/design/OAMP_TRANSPORT.md` と `kizuna-dream/docs/design/WEBSOCKET_EVENT_STREAM.md` を参照。

これらの実装は、仕様の準拠圧力テストです；いずれかがこのドラフトに対してクリーンに実装できない場合、ドラフトは v1.1 が最終化される前に改訂されます。
